# 세션 A 프롬프트: DirAcc 재현 검증 + 공정 평가 체계 구축 (exp019)

## 배경 및 목표

현재 최선 결과: MAE 8.39, DirAcc 0.40.
**목표: MAE < 5.0, DirAcc > 0.5.**
이를 위해 (1) DirAcc 0.55의 출처를 추적하고, (2) 공정한 평가 체계를 확립하고, (3) 근본적 성능 개선 방향을 탐색한다.

---

## 파트 1: DirAcc 0.55 재현 검증 (최우선)

### 배경
- exp011 Attn+Bottleneck: DirAcc **0.550**, cold MAE **67.71**
- exp016 G1 (residual, 16주): DirAcc **0.386**, cold MAE **8.77**
- exp018 최적: DirAcc **0.402**, MAE **8.39**

0.55와 0.40의 차이(0.15)는 매우 크다. 이것이 실제 성능 차이인지, 평가 방식 차이인지 반드시 규명해야 한다.

### 작업

**1-1: exp011의 DirAcc 계산 코드를 정확히 추적하라.**

exp011의 결과가 저장된 경로를 찾고, DirAcc을 계산한 코드를 찾아라.
다음을 확인하라:
- DirAcc의 정의: 주간 변화 방향(상승/하락/flat)의 일치율인가?
- flat 처리: flat(변화=0)을 어떻게 처리하는가? (flat==flat → 일치? 또는 flat 제외?)
- 평가 기간: 몇 주인가? 불완전한 주를 포함하는가?
- y_pred와 y_actual의 shape과 값 범위

**1-2: exp016/018의 DirAcc 계산 코드와 비교하라.**

exp016에서 사용한 `evaluate_weekly` 함수의 DirAcc 계산 로직을 찾아라.
exp011의 DirAcc 계산과 **라인 바이 라인**으로 비교하라.

구체적으로:
- 같은 함수를 사용하는가, 다른 함수인가?
- flat 처리 방식이 동일한가?
- 입력 데이터의 구조(wide vs long, 주 수, 아이템 수)가 동일한가?

**1-3: exp011의 원본 예측값으로 exp016의 평가 함수를 돌려라.**

exp011의 cold 예측값(Attn+Bottleneck)이 저장되어 있다면, 이를 exp016의 evaluate_weekly로 재평가하라.
- 같은 DirAcc이 나오면 → 평가 함수는 동일, 성능 차이가 실제
- 다른 DirAcc이 나오면 → 평가 함수의 차이가 원인

exp011의 예측값이 저장되어 있지 않다면, exp011의 학습 코드를 그대로 실행하여 예측값을 생성하라.

**1-4: 보고 내용**
- exp011과 exp016/018의 DirAcc 계산 방식 차이점 목록
- 동일 예측값에 대한 두 방식의 DirAcc 비교
- 0.55가 "올바른" 수치인지, 0.40이 "올바른" 수치인지 판정

## 파트 2: 공정 평가 체계 구축

### 배경
MAE와 DirAcc만으로는 방법론의 우수성을 충분히 보일 수 없다.
리뷰어가 "통계적으로 유의하지 않다"거나 "다른 지표로는 어떠냐"라고 물을 것이다.

### 2-1: M5 표준 평가 지표 도입

다음 지표를 **모든 모델**에 대해 계산하라:

| 지표 | 정의 | 특성 |
|------|------|------|
| MAE | mean absolute error | 현재 사용 중 |
| RMSE | root mean squared error | 큰 오차에 민감 |
| WRMSSE | weighted root mean squared scaled error | M5 공식 지표, 스케일 정규화 |
| MASE | mean absolute scaled error | naive 대비 상대 성능, cold-start에 적합 |
| DirAcc | directional accuracy | 방향 예측 |
| sMAPE | symmetric mean absolute percentage error | 비율 기반 |

MASE 계산: MASE = MAE / MAE_naive, 여기서 naive = 같은 카테고리 평균 (cold-start이므로 item-own lag 사용 불가)

### 2-2: 통계적 유의성 검증

Track B(10 seeds)와 각 competitor 간:
- MAE에 대한 Wilcoxon signed-rank test (paired, item별)
- 또는 bootstrap confidence interval for MAE difference
- p-value와 effect size 보고

## 파트 3: 성능 근본 개선을 위한 분석

### 배경
MAE 5 이하를 달성하려면 현재 8.39에서 40% 이상 개선해야 한다.
Head 최적화로는 불가능하다. Embedding이 담는 정보의 질을 올려야 한다.

### 3-1: 아이템별 오차 분석

cold 100개 각각에 대해:
- 평균 주간 실제 판매량
- 평균 주간 예측 판매량 (Track B 최적 모델)
- 아이템별 MAE
- 아이템 카테고리, 부서, 가격

이 데이터를 아이템별 MAE 내림차순으로 정렬하고, **상위 20개(MAE가 큰 아이템)**의 특성을 분석하라.
- 이 아이템들이 특정 카테고리에 집중되는가?
- 고판매 아이템인가 저판매 아이템인가?
- 가격대가 특이한가?

### 3-2: 오차 분해 — 스케일 오차 vs 방향 오차

각 아이템에 대해:
- 스케일 오차: |mean(pred) - mean(actual)| (평균 수준의 차이)
- 패턴 오차: MAE - 스케일 오차의 기여분 (변동 패턴의 차이)

이 분해를 통해 "MAE가 큰 아이템은 스케일이 안 맞아서인가, 변동 패턴이 안 맞아서인가"를 판단한다.

### 3-3: Competitor가 우리보다 잘하는 아이템 분석

lightgbm_proxy_lags와 Track B를 아이템별로 비교:
- lightgbm이 더 좋은 아이템은 몇 개인가?
- 그 아이템들의 공통 특성은?
- Track B가 더 좋은 아이템의 공통 특성은?

이 분석이 "어디를 개선하면 MAE를 크게 줄일 수 있는지"를 알려준다.
